Rechercher avec `find`

`find` sert à récupérer des documents précis dans une collection. C'est l'équivalent pratique du `SELECT ... WHERE` relationnel, adapté à des documents JSON/BSON.

  • On choisit une collection.
  • On écrit un filtre.
  • On limite les champs retournés avec une projection.
  • On trie, limite ou compte selon le besoin.

Comparer avec SQL

SQL

SELECT name, cuisine, price_tier
FROM restaurants
WHERE cuisine = 'Italian'
ORDER BY name ASC
LIMIT 10;

MongoDB

db.restaurants.find(
  { cuisine: "Italian" },
  { _id: 0, name: 1, cuisine: 1, price_tier: 1 }
).sort({ name: 1 }).limit(10)

La logique est proche, mais MongoDB manipule des documents : les champs peuvent être imbriqués, absents, ou contenir des tableaux.

Forme générale

db.collection.find(filtre, projection)
ÉlémentRôleExemple
`collection`Famille de documents interrogée.`restaurants`
`filtre`Conditions de sélection.`{ cuisine: "Italian" }`
`projection`Champs retournés.`{ _id: 0, name: 1 }`
db.restaurants.find(
  { cuisine: "Italian" },
  { _id: 0, name: 1, price_tier: 1, ratings: 1 }
)

Construire une requête pas à pas

Question métier : quels restaurants italiens sont abordables et bien notés ?

// 1. Filtrer une cuisine
db.restaurants.find({ cuisine: "Italian" })
// 2. Ajouter une contrainte de prix
db.restaurants.find({
  cuisine: "Italian",
  price_for_two: { $lte: 50 }
})
// 3. Ajouter une contrainte sur une note imbriquée
db.restaurants.find({
  cuisine: "Italian",
  price_for_two: { $lte: 50 },
  "ratings.overall": { $gte: 22 }
})

Projection : ne retourner que l'utile

Par défaut, MongoDB retourne tout le document, y compris `_id`. Une projection réduit la sortie et rend la lecture plus claire.

db.restaurants.find(
  {
    cuisine: "Italian",
    "ratings.overall": { $gte: 22 }
  },
  {
    _id: 0,
    name: 1,
    cuisine: 1,
    price_for_two: 1,
    "ratings.overall": 1
  }
)

SQL comparable : `SELECT name, cuisine, price_for_two, overall_rating ...`

Opérateurs de comparaison

OpérateurSensExemple
`$gt`Strictement supérieur`{ price_for_two: { $gt: 60 } }`
`$gte`Supérieur ou égal`{ "ratings.overall": { $gte: 23 } }`
`$lt`Strictement inférieur`{ price_for_two: { $lt: 40 } }`
`$lte`Inférieur ou égal`{ price_for_two: { $lte: 45 } }`
`$ne`Différent`{ price_tier: { $ne: "$$$$" } }`
db.restaurants.find({
  price_for_two: { $lte: 45 },
  "ratings.service": { $gte: 22 }
})

Appartenance et logique

`$in` et `$nin`

db.restaurants.find({
  price_tier: { $in: ["$", "$$"] }
})
db.restaurants.find({
  cuisine: { $nin: ["Steakhouse", "Seafood"] }
})

SQL comparable : `WHERE price_tier IN (...)`.

`$or` et AND implicite

db.restaurants.find({
  price_tier: { $ne: "$$$$" },
  $or: [
    { "ratings.food": { $gte: 24 } },
    { "ratings.service": { $gte: 24 } }
  ]
})

Plusieurs champs au même niveau signifient `AND`.

Documents imbriqués

MongoDB interroge un champ imbriqué avec la notation pointée. Les guillemets sont nécessaires autour du chemin.

db.restaurants.find({
  "ratings.food": { $gte: 24 },
  "ratings.decor": { $gte: 20 }
})
db.reviews.find({
  "scores.overall": { $gte: 23 },
  sentiment: "positive"
})

En SQL, ces informations seraient souvent réparties en colonnes ou dans une table reliée. Ici, elles vivent dans le document.

Tableaux : tags et opérateurs utiles

// Le tableau contient au moins cette valeur
db.restaurants.find({ tags: "top_food" })
// Le tableau contient toutes ces valeurs
db.restaurants.find({
  tags: { $all: ["top_food", "great_service"] }
})
// Le tableau contient au moins une des valeurs
db.restaurants.find({
  tags: { $in: ["fine_dining", "good_value"] }
})

Un index sur un tableau devient un index multikey. Il est utile, mais il faut éviter de multiplier les index multikey sans besoin clair.

Tri, limite, pagination, comptage

db.restaurants.find(
  {},
  { _id: 0, name: 1, cuisine: 1, "ratings.overall": 1 }
).sort({ "ratings.overall": -1 }).limit(10)
db.restaurants.find(
  {},
  { _id: 0, name: 1, cuisine: 1 }
).sort({ name: 1 }).skip(20).limit(10)
db.restaurants.countDocuments({ price_tier: "$$" })
db.restaurants.distinct("cuisine")

`skip` est simple pour apprendre, mais peut devenir coûteux sur de gros volumes. En production, on privilégie souvent une pagination par curseur.

À quoi sert un index ?

Sans index, MongoDB peut devoir parcourir toute la collection : c'est un `COLLSCAN`. Avec un index adapté, MongoDB parcourt une structure triée avant d'aller lire les documents utiles : c'est un `IXSCAN`.

SituationSans indexAvec index
Filtrer par `cuisine`Lecture de nombreux documents.Accès direct aux valeurs indexées.
Trier par `ratings.overall`Tri en mémoire après lecture.Lecture déjà ordonnée si l'index correspond.
Paginer des commandes récentesCoûteux sur gros volume.Parcours efficace par `restaurant_id` et date.
db.restaurants.createIndex({ "ratings.overall": -1 })
db.restaurants.createIndex({ cuisine: 1, price_tier: 1 })
db.orders.createIndex({ restaurant_id: 1, created_at: -1 })

Un index accélère certaines lectures, mais il coûte de l'espace disque et ralentit un peu les écritures, car MongoDB doit maintenir l'index à chaque insertion, modification ou suppression.

Vérifier avec `explain()`

On ne devine pas la performance : on observe le plan d'exécution. `explain("executionStats")` indique combien de documents et de clés d'index ont été examinés.

db.restaurants.find({
  cuisine: "French",
  price_tier: "$$"
}).explain("executionStats")
IndicateurInterprétation
`COLLSCAN`MongoDB parcourt la collection. À surveiller sur gros volume.
`IXSCAN`MongoDB utilise un index.
`totalDocsExamined`Nombre de documents lus.
`totalKeysExamined`Nombre d'entrées d'index parcourues.
`executionTimeMillis`Temps mesuré pour la requête.

Index simples, composés et multikey

Un index simple porte sur un champ. Un index composé porte sur plusieurs champs dans un ordre précis. Un index sur un tableau devient automatiquement multikey.

// Champ imbriqué
db.restaurants.createIndex({ "ratings.overall": -1 })

// Index composé : filtre cuisine + prix, puis tri possible par note
db.restaurants.createIndex({ cuisine: 1, price_tier: 1, "ratings.overall": -1 })

// Tableau tags : index multikey
db.restaurants.createIndex({ tags: 1 })

// Historique de commandes d'un restaurant
db.orders.createIndex({ restaurant_id: 1, status: 1, created_at: -1 })

Règle pratique : commencer l'index composé par les champs d'égalité fréquents, puis les champs de plage ou de tri. Exemple : `restaurant_id`, puis `status`, puis `created_at`.

Index géospatial : `2dsphere`

Le sandbox contient une petite collection `neighborhoods`. Chaque quartier possède un champ GeoJSON `center` avec un point `[longitude, latitude]`. Un index `2dsphere` est créé sur ce champ.

{
  name: "Midtown",
  borough: "Manhattan",
  center: {
    type: "Point",
    coordinates: [-73.9855, 40.7580]
  }
}
db.neighborhoods.createIndex({ center: "2dsphere" })

Cet index permet des requêtes géographiques, par exemple chercher les quartiers proches d'un point.

db.neighborhoods.find({
  center: {
    $near: {
      $geometry: { type: "Point", coordinates: [-73.9855, 40.7580] },
      $maxDistance: 5000
    }
  }
})

Attention : les coordonnées GeoJSON sont toujours dans l'ordre longitude puis latitude. Ici l'index géospatial sert surtout d'introduction, car les restaurants du dataset n'ont pas de coordonnées précises.

Limites de `find`

`find` est excellent pour lire des documents correspondant à des critères. Il devient moins adapté dès que la question demande de transformer les données.

Besoin`find` suffit ?Outil recommandé
Afficher les restaurants italiensOui`find`
Trier les meilleurs restaurantsOui`find().sort()`
Calculer une moyenne par cuisineNon`aggregate` + `$group`
Joindre commandes et restaurantsNon`aggregate` + `$lookup`
Créer une collection de KPINon`aggregate` + `$merge`

Message clé : `find` répond à “quels documents ?”. `aggregate` répond à “quels indicateurs ou transformations ?”.
