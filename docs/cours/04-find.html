Rechercher avec `find`

`find` sert à récupérer des documents précis dans une collection. C'est l'équivalent pratique du `SELECT ... WHERE` relationnel, adapté à des documents JSON/BSON.

  • On choisit une collection.
  • On écrit un filtre.
  • On limite les champs retournés avec une projection.
  • On trie, limite ou compte selon le besoin.

Comparer avec SQL

SQL

SELECT name, cuisine, price_tier
FROM restaurants
WHERE cuisine = 'Italian'
ORDER BY name ASC
LIMIT 10;

MongoDB

db.restaurants.find(
  { cuisine: "Italian" },
  { _id: 0, name: 1, cuisine: 1, price_tier: 1 }
).sort({ name: 1 }).limit(10)

La logique est proche, mais MongoDB manipule des documents : les champs peuvent être imbriqués, absents, ou contenir des tableaux.

Forme générale

db.collection.find(filtre, projection)
ÉlémentRôleExemple
`collection`Famille de documents interrogée.`restaurants`
`filtre`Conditions de sélection.`{ cuisine: "Italian" }`
`projection`Champs retournés.`{ _id: 0, name: 1 }`
db.restaurants.find(
  { cuisine: "Italian" },
  { _id: 0, name: 1, price_tier: 1, ratings: 1 }
)

Construire une requête pas à pas

Question métier : quels restaurants italiens sont abordables et bien notés ?

// 1. Filtrer une cuisine
db.restaurants.find({ cuisine: "Italian" })
// 2. Ajouter une contrainte de prix
db.restaurants.find({
  cuisine: "Italian",
  price_for_two: { $lte: 50 }
})
// 3. Ajouter une contrainte sur une note imbriquée
db.restaurants.find({
  cuisine: "Italian",
  price_for_two: { $lte: 50 },
  "ratings.overall": { $gte: 22 }
})

Projection : ne retourner que l'utile

Par défaut, MongoDB retourne tout le document, y compris `_id`. Une projection réduit la sortie et rend la lecture plus claire.

db.restaurants.find(
  {
    cuisine: "Italian",
    "ratings.overall": { $gte: 22 }
  },
  {
    _id: 0,
    name: 1,
    cuisine: 1,
    price_for_two: 1,
    "ratings.overall": 1
  }
)

SQL comparable : `SELECT name, cuisine, price_for_two, overall_rating ...`

Opérateurs de comparaison

OpérateurSensExemple
`$gt`Strictement supérieur`{ price_for_two: { $gt: 60 } }`
`$gte`Supérieur ou égal`{ "ratings.overall": { $gte: 23 } }`
`$lt`Strictement inférieur`{ price_for_two: { $lt: 40 } }`
`$lte`Inférieur ou égal`{ price_for_two: { $lte: 45 } }`
`$ne`Différent`{ price_tier: { $ne: "$$$$" } }`
db.restaurants.find({
  price_for_two: { $lte: 45 },
  "ratings.service": { $gte: 22 }
})

Appartenance et logique

`$in` et `$nin`

db.restaurants.find({
  price_tier: { $in: ["$", "$$"] }
})
db.restaurants.find({
  cuisine: { $nin: ["Steakhouse", "Seafood"] }
})

SQL comparable : `WHERE price_tier IN (...)`.

`$or` et AND implicite

db.restaurants.find({
  price_tier: { $ne: "$$$$" },
  $or: [
    { "ratings.food": { $gte: 24 } },
    { "ratings.service": { $gte: 24 } }
  ]
})

Plusieurs champs au même niveau signifient `AND`.

Documents imbriqués

MongoDB interroge un champ imbriqué avec la notation pointée. Les guillemets sont nécessaires autour du chemin.

db.restaurants.find({
  "ratings.food": { $gte: 24 },
  "ratings.decor": { $gte: 20 }
})
db.reviews.find({
  "scores.overall": { $gte: 23 },
  sentiment: "positive"
})

En SQL, ces informations seraient souvent réparties en colonnes ou dans une table reliée. Ici, elles vivent dans le document.

Expressions avec `$expr`

Dans une requête `find`, `$expr` permet d'utiliser une expression de calcul ou de comparaison entre champs du même document.

// food dépasse decor d'au moins 4 points
db.restaurants.find({
  $expr: {
    $gte: [
      { $subtract: ["$ratings.food", "$ratings.decor"] },
      4
    ]
  }
})

Question rapide : comment chercher les restaurants où l'écart entre `food` et `decor` est d'au moins 4 points, peu importe le sens de l'écart ?

Indice : on peut combiner `$subtract` avec `$abs` pour travailler en valeur absolue.

Voir une solution possible 
db.restaurants.find({
  $expr: {
    $gte: [
      { $abs: { $subtract: ["$ratings.food", "$ratings.decor"] } },
      4
    ]
  }
})

À retenir : `$expr` sert quand une condition ne compare pas seulement un champ à une valeur fixe, mais dépend d'un calcul ou d'une comparaison entre champs.

Tableaux : tags et opérateurs utiles

// Le tableau contient au moins cette valeur
db.restaurants.find({ tags: "top_food" })
// Le tableau contient toutes ces valeurs
db.restaurants.find({
  tags: { $all: ["top_food", "great_service"] }
})
// Le tableau contient au moins une des valeurs
db.restaurants.find({
  tags: { $in: ["fine_dining", "good_value"] }
})

À retenir : `$all` est un `AND` appliqué aux éléments d'un même tableau. Ici, `tags` doit contenir `top_food` ET `great_service`. Ce n'est pas un `$and` général : `$and` combine des conditions de requête, éventuellement sur plusieurs champs, alors que `$all` exprime précisément "toutes ces valeurs dans ce tableau".

On peut réécrire l'idée avec `$and` sur le même champ.

db.restaurants.find({
  $and: [
    { tags: "top_food" },
    { tags: "great_service" }
  ]
})

Mais `$all` est plus clair ici, car il exprime directement l'intention : vérifier que le tableau `tags` contient toutes les valeurs demandées.

Autre exemple avec `$and` : les conditions portent ici sur des champs différents.

db.restaurants.find({
  $and: [
    { cuisine: "French" },
    { price_tier: "$$" },
    { "ratings.overall": { $gte: 22 } }
  ]
})

Un index sur un tableau devient un index multikey. Il est utile, mais il faut éviter de multiplier les index multikey sans besoin clair.

Existence et type

`$exists` vérifie qu'un champ est présent. `$type` vérifie son type BSON. C'est utile sur une base documentaire, car tous les documents n'ont pas forcément exactement la même structure.

Exemple : repérer les restaurants pour lesquels `price_for_two` existe bien et peut être utilisé comme nombre.

db.restaurants.find({
  price_for_two: { $exists: true, $type: "number" }
})

On peut ensuite filtrer ou trier ce champ sans mélanger des documents où le prix serait absent ou stocké dans un mauvais type.

Expressions régulières : chercher dans du texte

Une expression régulière permet de rechercher un motif dans une chaîne de caractères. Dans nos exercices, on l'utilise surtout pour chercher un mot dans le nom d'un restaurant ou dans le texte d'un avis.

// Recherche insensible à la casse
db.restaurants.find({
  name: { $regex: "bistro", $options: "i" }
})

// Plusieurs mots possibles : cafe OU bistro OU ristorante
db.restaurants.find({
  name: { $regex: "cafe|bistro|ristorante", $options: "i" }
})

// Nom qui commence par cafe
db.restaurants.find({
  name: { $regex: "^cafe", $options: "i" }
})

`$regex` contient le motif recherché. `$options: "i"` ignore la casse. Le symbole `|` signifie "ou" dans le motif, et `^` signifie "début du texte". À ce niveau, l'objectif est seulement de savoir écrire une recherche textuelle simple.

Pour aller plus loin : documentation officielle MongoDB sur `$regex`.

Tri, limite, pagination, comptage

db.restaurants.find(
  {},
  { _id: 0, name: 1, cuisine: 1, "ratings.overall": 1 }
).sort({ "ratings.overall": -1 }).limit(10)
db.restaurants.find(
  {},
  { _id: 0, name: 1, cuisine: 1 }
).sort({ name: 1 }).skip(20).limit(10)
db.restaurants.countDocuments({ price_tier: "$$" })
db.restaurants.distinct("cuisine")

`skip` est simple pour apprendre, mais peut devenir coûteux sur de gros volumes. En production, on privilégie souvent une pagination par curseur.

Construire une requête métier

Les opérateurs isolés deviennent utiles quand on les combine pour répondre à une question formulée en français.

Une requête métier combine souvent un filtre principal, une projection, puis un tri ou une limite.

// Restaurants très bien notés mais abordables
db.restaurants.find(
  {
    "ratings.overall": { $gte: 22 },
    price_for_two: { $lte: 50 }
  },
  { _id: 0, name: 1, cuisine: 1, price_for_two: 1, "ratings.overall": 1 }
).sort({ "ratings.overall": -1, price_for_two: 1 })

// Avis détaillés négatifs et vérifiés
db.review_details.find({
  verified_visit: true,
  sentiment: "negative"
})

Ces exemples préparent la réflexion sur les index : les champs souvent utilisés dans les filtres ou les tris sont les premiers candidats à observer avec `explain()`.

À quoi sert un index ?

Quand une collection contient peu de documents, MongoDB peut parcourir toute la collection sans que cela se voie vraiment. Plus le volume augmente, plus certains filtres, tris ou historiques deviennent coûteux.

Un index sert à éviter de relire tous les documents quand MongoDB peut s'appuyer sur un champ déjà organisé pour la recherche. C'est proche de l'index d'un livre : on consulte une structure qui pointe vers les entrées utiles.

Important : un index ne réordonne pas physiquement les documents dans la collection. Il crée une structure séparée, triée par champ indexé, qui pointe vers les documents.

Collection restaurants :
[doc A] [doc B] [doc C]

Index sur ratings.overall :
18.5 -> doc C
20.7 -> doc A
24.0 -> doc B

Sans index, MongoDB peut devoir parcourir toute la collection : c'est un `COLLSCAN`. Avec un index adapté, MongoDB parcourt une structure triée avant d'aller lire les documents utiles : c'est un `IXSCAN`.

  • `COLLSCAN` = collection scan : MongoDB lit les documents un par un.
  • `IXSCAN` = index scan : MongoDB parcourt un index pour trouver plus vite les documents utiles.
SituationSans indexAvec index
Filtrer par `cuisine`Lecture de nombreux documents.Accès direct aux valeurs indexées.
Trier par `ratings.overall`Tri en mémoire après lecture.Lecture déjà ordonnée si l'index correspond.
Paginer des commandes récentesCoûteux sur gros volume.Parcours efficace par `restaurant_id` et date.
db.restaurants.createIndex({ "ratings.overall": -1 })
db.restaurants.createIndex({ cuisine: 1, price_tier: 1 })
db.orders.createIndex({ restaurant_id: 1, created_at: -1 })

Par défaut, un index n'impose pas l'unicité. Il sert seulement à accélérer certaines recherches ou certains tris.

db.restaurants.createIndex({ cuisine: 1 })

Cet index aide les requêtes sur `cuisine`, mais plusieurs restaurants peuvent avoir `cuisine: "Italian"`.

Pour interdire les doublons sur un champ, il faut créer un index unique.

db.restaurants.createIndex(
  { restaurant_id: 1 },
  { unique: true }
)

Avec cet index, MongoDB refuse deux documents qui auraient le même `restaurant_id`. Si une insertion crée un doublon, MongoDB renvoie une erreur de type `E11000 duplicate key error`.

  • `_id` est unique automatiquement.
  • Un index sans `{ unique: true }` accélère les requêtes, mais autorise les doublons.
  • Un index avec `{ unique: true }` accélère les requêtes et impose l'unicité.
  • `getIndexes()` permet de voir les index existants et de repérer ceux qui sont uniques.

Un index accélère certaines lectures, mais il coûte de l'espace disque et ralentit un peu les écritures, car MongoDB doit maintenir l'index à chaque insertion, modification ou suppression.

Vérifier avec `explain()`

On ne devine pas la performance : on observe le plan d'exécution. `explain("executionStats")` indique combien de documents et de clés d'index ont été examinés.

Exemple : on veut savoir comment MongoDB exécute une recherche sur la cuisine et le niveau de prix.

db.restaurants.find({
  cuisine: "French",
  price_tier: "$$"
}).explain("executionStats")

Si le résultat contient `COLLSCAN`, MongoDB parcourt la collection. On peut alors tester un index adapté aux champs filtrés.

db.restaurants.createIndex({ cuisine: 1, price_tier: 1 })

db.restaurants.find({
  cuisine: "French",
  price_tier: "$$"
}).explain("executionStats")

Après création de l'index, on cherche plutôt `IXSCAN` dans le plan. `explain()` retourne beaucoup d'informations : dans ce cours, on se concentre seulement sur quelques indicateurs pratiques.

  1. MongoDB lit-il la collection (`COLLSCAN`) ou un index (`IXSCAN`) ?
  2. Combien de documents sont examinés avec `totalDocsExamined` ?
  3. Combien de documents sont retournés avec `nReturned` ?
  4. Est-ce que `executionTimeMillis` baisse après un index pertinent ?
IndicateurInterprétation
`COLLSCAN`MongoDB parcourt la collection. À surveiller sur gros volume.
`IXSCAN`MongoDB utilise un index.
`totalDocsExamined`Nombre de documents lus.
`nReturned`Nombre de documents retournés par la requête.
`totalKeysExamined`Nombre d'entrées d'index parcourues.
`executionTimeMillis`Temps mesuré pour la requête.

Objectif pratique : pour une requête sélective, `totalDocsExamined` doit rester proche de `nReturned`, ou au moins très inférieur à la taille totale de la collection.

Index simples, composés et multikey

Un index simple porte sur un champ. Un index composé porte sur plusieurs champs dans un ordre précis. Un index sur un tableau devient automatiquement multikey.

// Champ imbriqué
db.restaurants.createIndex({ "ratings.overall": -1 })

// Index composé : filtre cuisine + prix, puis tri possible par note
db.restaurants.createIndex({ cuisine: 1, price_tier: 1, "ratings.overall": -1 })

// Tableau tags : index multikey
db.restaurants.createIndex({ tags: 1 })

// Historique de commandes d'un restaurant
db.orders.createIndex({ restaurant_id: 1, status: 1, created_at: -1 })

Un index composé est trié dans l'ordre des champs déclarés.

Restaurant`cuisine``price_tier`
AFrench`$$$`
BItalian`$$`
CFrench`$`
DJapanese`$$`
EFrench`$$`

Index `{ cuisine: 1, price_tier: 1 }` :

Ordre`cuisine``price_tier`Restaurant
1French`$`C
2French`$$`E
3French`$$$`A
4Italian`$$`B
5Japanese`$$`D

Index `{ price_tier: 1, cuisine: 1 }` :

Ordre`price_tier``cuisine`Restaurant
1`$`FrenchC
2`$$`FrenchE
3`$$`ItalianB
4`$$`JapaneseD
5`$$$`FrenchA

À retenir : les deux index utilisent les mêmes champs, mais ils ne rangent pas les entrées de la même manière. MongoDB peut donc les exploiter différemment selon les filtres et les tris.

Règle pratique : commencer l'index composé par les champs d'égalité fréquents, puis les champs de plage ou de tri. Exemple : `restaurant_id`, puis `status`, puis `created_at`.

Index géospatial : `2dsphere`

Le sandbox contient une petite collection `neighborhoods`. Chaque quartier possède un champ GeoJSON `center` avec un point `[longitude, latitude]`. Un index `2dsphere` est créé sur ce champ.

{
  name: "Midtown",
  borough: "Manhattan",
  center: {
    type: "Point",
    coordinates: [-73.9855, 40.7580]
  }
}
db.neighborhoods.createIndex({ center: "2dsphere" })

Cet index permet des requêtes géographiques, par exemple chercher les quartiers proches d'un point.

db.neighborhoods.find({
  center: {
    $near: {
      $geometry: { type: "Point", coordinates: [-73.9855, 40.7580] },
      $maxDistance: 5000
    }
  }
})

Attention : les coordonnées GeoJSON sont toujours dans l'ordre longitude puis latitude. Ici l'index géospatial sert surtout d'introduction, car les restaurants du dataset n'ont pas de coordonnées précises.

Limites de `find`

`find` est excellent pour lire des documents correspondant à des critères. Il devient moins adapté dès que la question demande de transformer les données.

Besoin`find` suffit ?Outil recommandé
Afficher les restaurants italiensOui`find`
Trier les meilleurs restaurantsOui`find().sort()`
Calculer une moyenne par cuisineNon`aggregate` + `$group`
Joindre commandes et restaurantsNon`aggregate` + `$lookup`
Créer une collection de KPINon`aggregate` + `$merge`

Message clé : `find` répond à “quels documents ?”. `aggregate` répond à “quels indicateurs ou transformations ?”.
