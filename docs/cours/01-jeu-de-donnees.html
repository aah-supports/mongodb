Jeu de données et structure des collections

Avant d'écrire des requêtes MongoDB, il faut comprendre précisément les documents disponibles : leur origine, leurs champs, leurs relations et les choix de modélisation.

Le sandbox travaille sur la base `nyc_food`. Elle mélange un dataset réel de notations de restaurants et des données générées pour simuler un contexte applicatif plus volumineux.

Source réelle : OpenIntro `nyc`

Le dataset réel contient 168 restaurants new-yorkais notés par Zagat. Il est importé depuis un CSV public.

Champ sourceSensType après import
`case`Identifiant de ligne dans la source.number
`restaurant`Nom du restaurant.string
`price`Prix estimé pour deux personnes.number
`food`Note nourriture sur 30.number
`decor`Note décor sur 30.number
`service`Note service sur 30.number
`east`Indicateur est/ouest de 5th Avenue.number

Le CSV est volontairement conservé dans une collection brute avant transformation. C'est une étape importante : importer des données ne suffit pas, il faut aussi choisir un modèle documentaire exploitable.

Le champ pivot le plus important est `restaurant_id`. Il relie `restaurants`, `reviews`, `orders`, `review_details` et `events`.

Collection brute : `nyc_restaurant_reviews_raw`

Cette collection conserve la donnée source après conversion des nombres. Elle sert à comparer le CSV initial avec le modèle documentaire final.

{
  source_case: 1,
  restaurant: "Daniella Ristorante",
  price: 43,
  food: 22,
  decor: 18,
  service: 20,
  east: 0,
  imported_at: ISODate("2026-05-30T...")
}
Usage pédagogiqueExemple

Repérer les conversions`price`, `food`, `decor`, `service` deviennent numériques.
Comprendre la transformationLa ligne brute devient un restaurant et une review agrégée.

Collection principale : `restaurants`

Un document représente un établissement. C'est la collection principale pour apprendre `find`, les projections, les champs imbriqués, les tableaux et les index.

{
  restaurant_id: "NYC-ZAGAT-0001",
  name: "Daniella Ristorante",
  cuisine: "Italian",
  location_area: "West of 5th Avenue",
  price_for_two: 43,
  price_tier: "$$",
  ratings: {
    food: 22,
    decor: 18,
    service: 20,
    overall: 20
  },
  rating_band: "very_good_to_excellent",
  tags: ["dinner"],
  source: {
    name: "OpenIntro nyc Zagat restaurant ratings",
    url: "https://raw.githubusercontent.com/OpenIntroStat/openintro/main/data-raw/nyc/nyc.csv",
    case: 1
  }
}

Collection `reviews` : avis agrégé

`reviews` représente la notation agrégée d'un restaurant. Elle est séparée de `restaurants` pour montrer une relation entre collections via `restaurant_id`.

{
  review_id: "ZAGAT-0001",
  restaurant_id: "NYC-ZAGAT-0001",
  reviewed_at: ISODate("2025-01-01T00:00:00Z"),
  reviewer_type: "local",
  scores: {
    food: 22,
    decor: 18,
    service: 20,
    overall: 20
  },
  price_for_two: 43,
  sentiment: "positive",
  highlights: ["dinner"],
  source: {
    name: "OpenIntro nyc Zagat restaurant ratings",
    case: 1
  }
}

Cette collection sert à pratiquer les champs imbriqués `scores.*`, les filtres par `sentiment`, puis les jointures avec `$lookup`.

Collection `neighborhoods`

`neighborhoods` est une petite collection pédagogique créée à l'initialisation. Elle montre qu'une base MongoDB peut contenir plusieurs familles de documents.

{
  name: "Midtown",
  borough: "Manhattan",
  center: {
    type: "Point",
    coordinates: [-73.9855, 40.7580]
  }
}

Un index `2dsphere` est créé sur `center`. Cette collection peut servir à discuter des données géographiques, même si le fil rouge principal reste centré sur les restaurants et avis.

Import de secours sans réseau

Si le téléchargement du CSV ne fonctionne pas, le dépôt contient un snapshot JSON dans `data/nyc-food-seed.json`. Il permet de charger les collections de base en une commande `mongosh`, sans appel externe.

cd sandbox-mongodb
docker compose up -d mongodb
docker compose exec mongodb mongosh \
  "mongodb://root:rootpass@localhost:27017/nyc_food?authSource=admin" \
  /scripts/import-seed-json.js

Cette commande remplit `nyc_restaurant_reviews_raw`, `restaurants`, `reviews` et `neighborhoods`, puis recrée les index utiles. Pour les collections volumineuses, lancer ensuite `generate-volume.js`.

Collection `orders`

Chaque document représente une commande simulée, rattachée à un restaurant par `restaurant_id`.

{
  order_id: "BULK-ORD-000000001",
  restaurant_id: "NYC-ZAGAT-0001",
  created_at: ISODate("2025-01-01T00:00:00Z"),
  channel: "web",
  status: "paid",
  amount: 42.35,
  items_count: 4,
  customer: {
    loyalty_tier: "silver",
    zipcode: "10001"
  }
}

Champs importants : `restaurant_id`, `created_at`, `status`, `channel`, `amount`, `customer.loyalty_tier`.

Collection `review_details`

Chaque document simule un avis client détaillé. Cette collection complète `reviews`, qui reste une notation agrégée issue du dataset réel.

{
  review_detail_id: "REV-DETAIL-000000001",
  restaurant_id: "NYC-ZAGAT-0001",
  reviewed_at: ISODate("2025-01-01T00:00:00Z"),
  rating: 4.7,
  sentiment: "positive",
  visit_reason: "business_lunch",
  channel: "mobile",
  verified_visit: true,
  helpful_votes: 12,
  text: "Great food and attentive service."
}

Elle sert à filtrer sur `verified_visit`, `rating`, `sentiment`, puis à agréger par restaurant ou raison de visite.

Collection `events`

Chaque document représente une action applicative simulée. C'est la collection la plus volumineuse du sandbox.

{
  event_id: "EVT-0000000001",
  restaurant_id: "NYC-ZAGAT-0001",
  event_type: "restaurant_view",
  occurred_at: ISODate("2025-01-01T00:00:00Z"),
  session_id: "S-123",
  device: "ios",
  latency_ms: 184
}

Elle sert à discuter des événements, des séries temporelles simples, des tris par date et des index sur `event_type` ou `restaurant_id`.

Relations entre collections

MongoDB ne force pas les clés étrangères comme une base relationnelle, mais le modèle applicatif utilise des identifiants communs.

restaurants.restaurant_id
  -> reviews.restaurant_id
  -> orders.restaurant_id
  -> review_details.restaurant_id
  -> events.restaurant_id
RelationUtilisation
`orders` -> `restaurants`Chiffre d'affaires avec nom, cuisine et note.
`review_details` -> `restaurants`Avis détaillés enrichis avec les infos du restaurant.
`events` -> `restaurants`Analyse d'activité par établissement.

Dans MongoDB, ces relations sont exploitées avec `$lookup` dans les pipelines d'agrégation.
