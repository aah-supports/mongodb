Utilisateurs, rôles et droits

Comprendre pourquoi une application ne doit pas utiliser un compte administrateur et comment limiter les privilèges MongoDB.

Les comptes du sandbox

Le sandbox contient un compte administrateur et un compte applicatif.

root / rootpass        # administrateur MongoDB
student / studentpass # compte applicatif nyc_food

`root` sert à administrer. `student` sert à travailler dans la base `nyc_food` avec le rôle `readWrite`.

Authentification et autorisation

L'authentification répond à la question : qui es-tu ?

L'autorisation répond à la question : as-tu le droit d'exécuter cette action ?

mongodb://root:rootpass@localhost:27017/nyc_food?authSource=admin
  • `nyc_food` est la base cible.
  • `authSource=admin` est la base où l'utilisateur `root` est défini.
  • Un utilisateur authentifié peut quand même manquer de droits.

Rôles intégrés utiles

  • `read` : lire les données d'une base.
  • `readWrite` : lire et écrire les données d'une base.
  • `dbAdmin` : administrer une base, index et statistiques.
  • `userAdmin` : gérer les utilisateurs d'une base.
  • `root` : tous les droits sur l'instance.

Une application doit recevoir le plus petit niveau de droit suffisant.

Utilisateur `read`

docker compose exec mongodb mongosh \
  "mongodb://root:rootpass@localhost:27017/nyc_food?authSource=admin"
db.createUser({
  user: "alan",
  pwd: "pass",
  roles: [{ role: "read", db: "nyc_food" }]
})
docker compose exec mongodb mongosh \
  "mongodb://alan:pass@localhost:27017/nyc_food"
mongosh -u alan -p pass \
  --authenticationDatabase nyc_food \
  nyc_food

Utilisateur `readWrite`

`readWrite` permet de lire et modifier les données d'une base, sans devenir administrateur.

db.createUser({
  user: "api_writer",
  pwd: "apiwriterpass",
  roles: [{ role: "readWrite", db: "nyc_food" }]
})
db.restaurants.findOne()
db.reviews.insertOne({
  restaurant_id: "NYC-ZAGAT-0001",
  source: "manual-test"
})

Ce compte peut écrire des documents, mais il ne devrait pas pouvoir créer d'autres utilisateurs.

Droits plus fins

Les rôles intégrés sont pratiques mais parfois trop larges. Un rôle personnalisé peut limiter les actions par collection.

db.createRole({
  role: "reviews_writer",
  privileges: [
    {
      resource: { db: "nyc_food", collection: "restaurants" },
      actions: ["find"]
    },
    {
      resource: { db: "nyc_food", collection: "reviews" },
      actions: ["find", "insert", "update"]
    }
  ],
  roles: []
})

Assigner ou retirer un rôle

db.createUser({
  user: "api_reviews",
  pwd: "apireviewspass",
  roles: [{ role: "reviews_writer", db: "nyc_food" }]
})
db.grantRolesToUser("alan", [
  { role: "readWrite", db: "nyc_food" }
])

db.revokeRolesFromUser("alan", [
  { role: "readWrite", db: "nyc_food" }
])

Tester les droits

Le test le plus clair consiste à exécuter une action autorisée, puis une action interdite.

db.restaurants.findOne()
db.restaurants.insertOne({
  restaurant_id: "TEST-READONLY",
  name: "Read Only Test"
})

Avec `alan`, la lecture doit réussir et l'écriture doit échouer.

Lien avec l'API Express

L'API du sandbox utilise le compte `student`, pas le compte `root`.

MONGODB_URI: mongodb://student:studentpass@mongodb:27017/nyc_food
  • L'application n'a pas les droits administrateur.
  • Les droits sont limités à `nyc_food`.
  • Le compte correspond au besoin applicatif.

À retenir

  • Ne pas utiliser `root` dans une application.
  • Créer un utilisateur dédié à chaque application.
  • Donner uniquement les droits nécessaires.
  • Utiliser un rôle personnalisé si `readWrite` est trop large.
  • Tester une action autorisée et une action interdite.
  • Stocker les URI de connexion dans des variables d'environnement.
